# ضمان انعكاس التعديلات القادمة والنشر التلقائي

## هل التعديلات القادمة ستنعكس تلقائياً؟

**نعم**، طالما التالي ثابت ولا يُغيّر:

| المكوّن | أين | لا تغيّره |
|---------|-----|-----------|
| السيكريتان | GitHub → Settings → Secrets → Actions | `DEPLOY_WEBHOOK_URL` و `DEPLOY_WEBHOOK_TOKEN` |
| الـ Cron | السيرفر (أمر `crontab -e`) | السطر الذي يشغّل النشر عند وجود العلم |
| ملف الرمز | السيرفر: `public_html/.deploy-token` | صلاحياته 640، ولا تحذفه |
| الويب هوك | السيرفر: `public_html/deploy-webhook.php` | يبقى كما هو (يُحدَّث مع كل نشر من `deploy_public_html`) |
| الـ Workflow | المستودع: `.github/workflows/check-and-build.yml` | لا تحذف job `deploy-trigger` |
| سكربت النشر | المستودع: `scripts/deploy-on-server.sh` | التعديل عليه فقط عند تغيير آلية النشر |

**ما تفعله أنت عند أي تعديل واجهة:**

1. تعديل في **`src/`** (أو أي مصدر للواجهة).
2. تشغيل **`npm run build`**.
3. **`git add src deploy_public_html server/public`** (وأي ملفات غيّرتها).
4. **`git commit -m "وصف"`** ثم **`git push origin main`**.

بعد الـ push، الـ workflow يعمل → الويب هوك يُستدعى → السيرفر يضع العلم → الـ cron خلال دقيقة يسحب وينسخ → الموقع يتحدّث. لا أوامر يدوية على السيرفر.

---

## هل الأوامر في المحلي والسيرفر تتأثر بتعديلات قادمة؟

**لا.** آلية النشر مبنية على ملفات ثابتة:

- **محلي:** التعديلات العادية في `src/` أو المحتوى لا تغيّر أوامر النشر. الأوامر مضبوطة في:
  - **`.github/workflows/check-and-build.yml`**
  - **`scripts/deploy-on-server.sh`**
  - **`script/build.ts`**
  
  ما لم تُعدّل هذه الملفات عمداً، طريقة النشر (الأوامر) تبقى كما هي.

- **سيرفر:** الـ **cron** مُسجّل في نظام التشغيل (crontab)، و**لا يعتمد على أي طرفية مفتوحة**. أي تعديل في الكود (محتوى الموقع) يُسحب عبر `git pull` داخل السكربت؛ لا يغيّر أوامر الـ cron أو سكربت النشر إلا إذا غيّرتها أنت على السيرفر أو في المستودع.

---

## هل إغلاق الطرفية يلغي التعديلات التلقائية؟

**لا.** إغلاق الطرفية (محلي أو SSH) **لا يلغي** النشر التلقائي:

- **GitHub Actions** يعمل على سيرفرات GitHub، وليس على جهازك. بعد الـ push، الـ workflow يشتغل بغض النظر عن طرفيتك.
- **الـ Cron** على السيرفر يعمل كخدمة في الخلفية. لا يحتاج طرفية مفتوحة؛ يعمل كل دقيقة ويبحث عن ملف العلم ويشغّل السكربت.

التعديلات التي انعكست على الموقع (وملفات النشر على السيرفر) تبقى حتى يأتي نشر جديد يغيّرها.

---

## ملخص سريع

| سؤال | جواب |
|------|------|
| كيف أضمن أن التعديلات القادمة ستنعكس؟ | لا تحذف السيكريتين ولا الـ cron ولا `.deploy-token`؛ عند كل تعديل واجهة: build ثم commit + push (مع `deploy_public_html`). |
| هل أوامر المحلي/السيرفر تتأثر؟ | لا، إلا إذا عدّلت أنت ملفات الـ workflow أو سكربت النشر أو الـ cron. |
| إغلاق الطرفية يلغي التلقائي؟ | لا. الـ workflow يعمل على GitHub، والـ cron يعمل على السيرفر دون حاجة لطرفية مفتوحة. |

بهذا يكون عملنا مضبوطاً والتعديلات القادمة ستنعكس تلقائياً ما دامت الإعدادات أعلاه لم تُغيّر.
