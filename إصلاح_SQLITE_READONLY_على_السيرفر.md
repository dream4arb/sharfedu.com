# إصلاح خطأ: SQLITE_READONLY_DBMOVED (تسجيل الدخول لا يعمل)

عند ظهور **"SQLITE_READONLY_DBMOVED: attempt to write a readonly database"** في صفحة تسجيل الدخول، السبب أن تطبيق Node على السيرفر لا يستطيع **الكتابة** على ملف قاعدة البيانات.

---

## الحل على السيرفر (خطوات تنفيذية)

### 1) التأكد من مسار قاعدة البيانات

اتصل بـ SSH وانتقل لمجلد تشغيل التطبيق (مثل `git_repo` أو `node_app`)، ثم:

```bash
# من يشغّل Node؟
ps aux | grep node

# أين ملف sqlite.db؟
ls -la sqlite.db
# أو
ls -la .env
cat .env | grep DATABASE
```

### 2) ضبط صلاحيات الكتابة

يجب أن يملك الملف والمجلد **نفس المستخدم** الذي يشغّل Node (مثلاً `master` على Cloudways):

```bash
# استبدل /المسار/إلى/مجلد/التطبيق بالمسار الفعلي
chmod 755 /المسار/إلى/مجلد/التطبيق
chmod 664 /المسار/إلى/مجلد/التطبيق/sqlite.db
chown المستخدم:المجموعة /المسار/إلى/مجلد/التطبيق/sqlite.db
```

لمعرفة المستخدم: انظر أول عمود من ناتج `ps aux | grep node`.

### 3) استخدام مسار مطلق في .env

في ملف `.env` على السيرفر:

```bash
# استخدم مساراً مطلقاً لملف القاعدة (تجنباً لـ READONLY_DBMOVED)
DATABASE_URL=file:/المسار/الكامل/إلى/sqlite.db
```

**مثال على Cloudways:**

```bash
DATABASE_URL=file:/home/master/applications/اسم_التطبيق/app/sqlite.db
```

استبدل `اسم_التطبيق` و`app` حسب مسار تطبيقك الفعلي.

### 4) تجنّب مجلد يتم استبداله عند النشر

إذا كان `sqlite.db` داخل مجلد يُحدَّث عند كل `git pull` أو نشر، انقله لمجلد ثابت (مثل `app_private` أو مجلد بيانات) وضبط `DATABASE_URL` على المسار الجديد.

### 5) إعادة تشغيل التطبيق

بعد تعديل الصلاحيات و`.env`:

```bash
pm2 restart all
# أو أوقف عملية Node ثم شغّلها من جديد
```

---

## ملخص سريع

| الخطوة | الإجراء |
|--------|---------|
| 1 | التأكد من مكان `sqlite.db` ومن المستخدم الذي يشغّل Node |
| 2 | `chmod 664` لملف القاعدة و`chown` للمستخدم الصحيح |
| 3 | تعيين `DATABASE_URL=file:/مسار/مطلق/sqlite.db` في `.env` |
| 4 | إعادة تشغيل Node (مثلاً `pm2 restart all`) |

بعد ذلك جرّب تسجيل الدخول من الموقع مرة أخرى.
