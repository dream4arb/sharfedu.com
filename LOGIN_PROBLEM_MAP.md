# أين تكمن مشكلة الدخول؟

مسار طلب تسجيل الدخول من المتصفح حتى قاعدة البيانات، ونقاط الفشل المحتملة:

---

## 1. الواجهة (المتصفح → طلب الدخول)

| الملف | الدور |
|-------|--------|
| `src/pages/Login.tsx` | يرسل `POST` إلى `getApiUrl("/api/login")` مع `email` و `password` و `credentials: "include"` |

**أين قد تفشل:**
- إذا كان **VITE_API_BASE** مضبوطاً في البناء (مثلاً `/api-proxy.php`) والموقع لا يستخدم بروكسي PHP، الطلب يذهب لمسار خاطئ → **404 أو خطأ**.
- إذا الطلب يذهب لمسار صحيح لكن **لا يُرسل معه الكوكيز لاحقاً** (نفس النطاق و CORS صحيح)، الجلسة لا تُربط بالمتصفح.

**التشخيص:** افتح أدوات المطور (F12) → تبويب Network → جرّب الدخول → انظر طلب `POST .../api/login`:
- إن ظهر **404** أو **503** → المشكلة قبل وصول الطلب لـ Node (انظر 2 و 3).
- إن ظهر **200** وبدنه `{ "user": ... }` لكن الصفحة لا "تسجّل دخول" → غالباً مشكلة جلسة/كوكي (انظر 5).

---

## 2. البوابة (.htaccess على السيرفر)

| الملف | الدور |
|-------|--------|
| `.htaccess` داخل مجلد الدومين (public_html) | إعادة توجيه أي طلب يبدأ بـ `/api` إلى `http://localhost:5000/api/...` |

**أين قد تفشل:**
- ملف **.htaccess** غير موجود في مجلد الدومين، أو القواعد لا تطبّق → الطلب لا يصل لـ Node → **404** (أو Apache يرد بصفحة ثابتة).
- **mod_proxy** أو **mod_proxy_http** غير مفعّلين على الاستضافة → إعادة التوجيه تفشل → **500 أو 503**.

**التشخيص:** افتح من المتصفح:
- `https://sharfedu.com/api/health`
  - إن ظهر `{"ok":true,"service":"sharfedu-api"}` → البوابة و Node يعملان.
  - إن ظهر **503** أو **لا يمكن الوصول** → إما Node غير مشغّل (3) أو البوابة لا تعمل (2).

---

## 3. تطبيق Node (السيرفر على المنفذ 5000)

| الملف | الدور |
|-------|--------|
| `server/index.ts` | تحميل `.env` وتشغيل Express على `PORT=5000` |
| `server/routes.ts` | تسجيل مسارات الـ API ومن بينها استدعاء `setupAuth` و `registerAuthRoutes` |
| `server/auth/localAuth.ts` | تعريف `POST /api/login` و Passport (local + Google) والجلسة |

**أين قد تفشل:**
- تطبيق **Node غير مشغّل** على المنفذ 5000 → كل طلبات `/api` تعطي **503**.
- ملف **.env** غير موجود أو غير مقروء (مثلاً من مجلد خاطئ) → `DATABASE_URL` أو مفاتيح جوجل ناقصة → خطأ عند تنفيذ الدخول أو جوجل.
- ترتيب الـ routes: إذا تم تقديم **ملفات ثابتة** (static) قبل مسارات الـ API وقد تُخدم طلب `/api/login` كملف → نادر؛ عندنا الـ API يُسجّل أولاً.

**التشخيص:** من السيرفر (SSH):
- `curl -s http://127.0.0.1:5000/api/health`  
  إن رجع `{"ok":true,...}` فـ Node يعمل.
- `curl -s -X POST http://127.0.0.1:5000/api/login -H "Content-Type: application/json" -d '{"email":"test@test.com","password":"123456"}'`  
  إن رجع **401** مع رسالة عربية → الطلب يصل لـ Node ومنطق الدخول يعمل (الخطأ "بريد/كلمة مرور") وليس البنية.

---

## 4. قاعدة البيانات وملف المستخدمين

| الملف | الدور |
|-------|--------|
| `server/db.ts` | يقرأ `DATABASE_URL` من `.env` ويتصل بـ SQLite |
| `server/auth/storage.ts` | `getUserByEmail` و `getUser` من جدول `users` |
| `shared/models/auth.ts` | تعريف جدول `users` (email, password, …) |

**أين قد تفشل:**
- **DATABASE_URL** خاطئ أو مسار ملف SQLite غير موجود → استثناء عند أول استعلام → **500** أو رسالة خطأ في سجلات Node.
- جدول **users** غير موجود (لم يُنفَّذ migration أو push) → استثناء من Drizzle → **500**.
- المستخدم غير موجود أو **كلمة المرور** لا تطابق الـ hash في الجدول → Passport يرجع `user: false` → **401** (هذا سلوك صحيح وليس عطل بنية).

**التشخيص:** من سجلات Node (مثلاً `pm2 logs`):
- إن ظهر `[Auth] Login error:` أو استثناء من `db` أو `storage` → المشكلة في الاتصال بقاعدة البيانات أو الجدول.
- إن ظهر `[Auth] Login failed:` مع رسالة عربية → الطلب وصل والـ DB تعمل، والسبب إما مستخدم غير موجود أو كلمة مرور خاطئة.

---

## 5. الجلسة والكوكي (بعد نجاح الدخول)

| الملف | الدور |
|-------|--------|
| `server/auth/localAuth.ts` | `getSession()`: تخزين الجلسة في SQLite، كوكي `connect.sid` مع `secure`, `sameSite: "lax"` |
| `server/index.ts` | `app.set("trust proxy", 1)` في `setupAuth` حتى يعتمد Express على `X-Forwarded-Proto` |

**أين قد تفشل:**
- في الإنتاج **cookie.secure = true**؛ إذا الموقع يُخدم عبر **HTTP** بدل **HTTPS**، المتصفح لا يرسل الكوكي → الجلسة لا تُربط.
- **نطاق الدومين** مختلف بين الصفحة وطلب الـ API (مثلاً www vs غير www) → الكوكي لا يُرسل.
- **trust proxy** غير مضبوط أو البروكسي لا يمرّر `X-Forwarded-Proto: https` → Express يظن الاتصال http فيرفض أو يربط الجلسة بشكل خاطئ.

**التشخيص:** بعد طلب دخول ناجح (200 و `{ "user": ... }`):
- في أدوات المطور → Application (أو Storage) → Cookies → تحقق من وجود **connect.sid** لـ `https://sharfedu.com`.
- إن لم يظهر الكوكي → المشكلة في إعداد الجلسة أو النطاق أو secure/proxy.

---

## ملخص: أين تكمن المشكلة؟

| إن كان يحدث... | فالمشكلة غالباً في... |
|-----------------|------------------------|
| 404 أو 503 لـ `/api/login` أو `/api/health` | البوابة (.htaccess) أو Node غير مشغّل أو .env |
| 200 من `/api/login` لكن الصفحة لا "تسجّل دخول" | الجلسة/كوكي (secure، نطاق، trust proxy) |
| 401 من `/api/login` مع رسالة عربية | البريد أو كلمة المرور (أو عدم وجود المستخدم في الـ DB) |
| 500 من `/api/login` أو رسالة خطأ في السجلات | قاعدة البيانات (مسار، جدول users، أو استثناء في الكود) |

الخطوة العملية الأولى: افتح **https://sharfedu.com/api/health** واذكر ماذا يظهر (200 + JSON أو رقم خطأ)، ثم نحدد الخطوة التالية بدقة.
