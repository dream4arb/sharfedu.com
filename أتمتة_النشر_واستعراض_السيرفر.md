# أتمتة النشر واستعراض السيرفر

## هيكل السيرفر (مطابق لما على Cloudways)

تحت مسار التطبيق **SERVER_APP_ROOT** (مثل `/home/894422.cloudwaysapps.com/cmkdrtgqcv`) توجد:

| المجلد على السيرفر | الوظيفة |
|---------------------|---------|
| **git_repo** | المستودع: الكود، `dist`، `deploy_public_html`، `server`، `src`، `node_modules`... (يُشغَّل منه Node: `node dist/index.cjs`) |
| **public_html** | جذر الويب الذي يخدمه Apache (الواجهة والملفات الثابتة — يُنسخ إليه محتوى `deploy_public_html` بعد كل نشر) |
| **conf** | إعدادات Cloudways |
| **logs** | السجلات (مثل `app.log`) |
| **private_html**, **tmp**, **.ssh**, **ssl** | مجلدات النظام |

سكربت **deploy-on-server.sh** يُشغَّل من داخل **git_repo**؛ ينسخ واجهة الموقع إلى **public_html** ثم يعيد تشغيل تطبيق Node من **git_repo**.

## 1) استعراض كل الملفات والمجلدات على السيرفر (لمطابقة المحلي وتجنب الازدواجية)

### الإعداد (مرة واحدة)

1. **نسخ ملف إعداد السيرفر:**
   ```bash
   copy .env.server.example .env.server
   ```
   (أو يدوياً: أنشئ ملفاً اسمه `.env.server` في جذر المشروع.)

2. **فتح `.env.server` وتعديل القيم:**
   - **SERVER_SSH:** مستخدم SSH وعنوان السيرفر (من لوحة Cloudways → Application → SSH).
     - مثال: `master_nyrmduupwf@server-12345.cloudwaysapps.com`
   - **SERVER_APP_ROOT:** مسار جذر التطبيق على السيرفر.
     - مثال: `/home/894422.cloudwaysapps.com/cmkdrtgqcv`

ملف `.env.server` لا يُرفع إلى Git (مضاف في `.gitignore`).

### تنفيذ أمر الاستعراض

من جذر المشروع **sharfedu.com**:

**Windows (PowerShell):**
```powershell
npm run server:list
```
أو مباشرة:
```powershell
.\scripts\list-server.ps1
```

**لحفظ القائمة في ملف (للمقارنة لاحقاً):**  
الناتج يُحفظ تلقائياً في `server-files-list.txt` في جذر المشروع. لمقارنة المحلي بالسيرفر: شغّل البناء محلياً (`npm run build`)، ثم قارن محتويات `deploy_public_html` و `dist` مع ما يظهر في القائمة على السيرفر.

**بدون حفظ (عرض فقط):**
```powershell
.\scripts\list-server.ps1 -NoSave
```

**Linux / Mac / Git Bash:**
```bash
export SERVER_SSH="master_xxx@server-xxx.cloudwaysapps.com"
export SERVER_APP_ROOT="/home/894422.cloudwaysapps.com/cmkdrtgqcv"
./scripts/list-server.sh
```
أو بعد تعبئة `.env.server` يمكنك استخدامه من السكربت (حسب دعم الـ source في بيئتك).

## 2) أتمتة الرفع: تعديل محلي → GitHub → السيرفر

### المسار التلقائي الحالي

1. **محلي:** تعديل الملفات ثم تشغيل أمر النشر (انظر أدناه).
2. **GitHub:** يتم الـ push إلى فرع `main`.
3. **GitHub Actions:**
   - **Check & Build:** يتحقق من عدم وجود `.env` في المستودع، ثم يشغّل `npm run build:upload`.
   - **Deploy to Cloudways:** يطلب من Cloudways تنفيذ **Git Pull** على السيرفر.
4. **السيرفر (Cloudways):** الـ workflow **يتصل بالسيرفر عبر SSH** ويشغّل سكربت النشر تلقائياً (سحب التحديثات، نسخ الواجهة إلى public_html، إعادة تشغيل Node). **لا حاجة لفتح SSH يدوياً** — انظر **النشر_التلقائي_عبر_SSH.md** لإعداد الـ Secrets مرة واحدة.

### أمر النشر من المحلي (رفع تلقائي إلى GitHub ثم السيرفر)

من جذر المشروع:

**Windows (PowerShell):**
```powershell
npm run deploy
```
أو مع رسالة commit مخصصة:
```powershell
.\scripts\deploy.ps1 "إصلاح صفحة تسجيل الدخول"
```

**ماذا يفعل السكربت:**
- يضيف كل التغييرات (`git add -A`).
- ينشئ commit بالرسالة المعطاة (أو الافتراضية: "deploy: تحديثات من المحلي").
- يرفع إلى `origin main` (`git push origin main`).

بعد الـ push تعمل تلقائياً:
- **Check & Build** (التحقق من عدم رفع `.env` ثم البناء).
- **Deploy to Cloudways** (اتصال SSH بالسيرفر وتشغيل سكربت النشر: سحب التحديثات → نسخ الواجهة إلى public_html → إعادة تشغيل التطبيق).

### متطلبات النشر التلقائي (مرة واحدة)

يجب إعداد **أربعة Secrets** في GitHub وإضافة **مفتاح SSH عام** على السيرفر حتى يعمل النشر التلقائي دون دخول SSH. الخطوات الكاملة في **النشر_التلقائي_عبر_SSH.md**.

**الـ Secrets المطلوبة:**

| الـ Secret | الغرض |
|-----------|--------|
| `SSH_PRIVATE_KEY` | المفتاح الخاص SSH (محتوى ملف المفتاح كاملاً) |
| `SERVER_SSH_HOST` | عنوان السيرفر (مثل `server-xxx.cloudwaysapps.com`) |
| `SERVER_SSH_USER` | مستخدم SSH (مثل `master_xxx`) |
| `SERVER_GIT_REPO_PATH` | المسار الكامل لمجلد المستودع (مثل `.../git_repo`) |

## ملخص الأوامر

| الهدف | الأمر |
|--------|--------|
| استعراض كل الملفات/المجلدات على السيرفر | `npm run server:list` أو `.\scripts\list-server.ps1` |
| حفظ القائمة في ملف للمقارنة | يُحفظ تلقائياً في `server-files-list.txt` |
| رفع التعديلات إلى GitHub ثم النشر على السيرفر | `npm run deploy` أو `.\scripts\deploy.ps1 "وصف التعديل"` |

بهذا يصبح أي تعديل محلي يُرفع بأمر واحد إلى GitHub، ومن GitHub يُسحب تلقائياً إلى السيرفر (بعد إعداد Cloudways و GitHub Secrets). وأمر استعراض السيرفر يساعد في التأكد من أن المحتوى على السيرفر مطابق لما تريده محلياً وتجنب الازدواجية أو الأخطاء.
